# 第8章　検索の重み付け ― boost・multi_match・function_score

> **フェーズ2「検索の改善」スタート**
> フェーズ1では「検索を動かす」ところまで作りました。第7章で見たとおり、Elasticsearch は BM25 によって関連度スコアを**自動計算**してくれます。しかし現実の検索では「商品名の一致は説明文より重視したい」「在庫のある商品を上位に出したい」「価格や新しさも順位に反映したい」といった**ビジネス都合**が出てきます。フェーズ2はこの「自動スコアを、意図どおりに曲げる」技術から始めます。

---

## 学習目標

この章を終えると、次のことができるようになります。

- クエリ時 `boost` を使い、特定の条件や語句の寄与を相対的に強める／弱める
- `multi_match` で複数フィールドを一度に検索し、`best_fields` / `most_fields` / `cross_fields` の違いを説明して使い分ける
- フィールド単位の重み付け（`name^3` のキャレット記法）を使う
- `function_score` でスコアに関数（`weight` / `field_value_factor` / 減衰関数 / `random_score`）を適用する
- `score_mode` と `boost_mode` の役割を区別し、関数スコアと BM25 スコアの合成方法を制御する
- 上記3つを組み合わせて、実務的な「関連度＋ビジネスルール」の検索を書く

## 8.1　なぜ「重み付け」が必要か

第7章で学んだとおり、BM25 スコアは「その語がこの文書でどれだけ特徴的か（TF・IDF）」を数学的に評価します。これは**言語的な関連度**としては優秀ですが、次のような**業務要件**は表現できません。

- 商品名の一致は、説明文の一致より重要だ（**フィールドの重要度**）
- 在庫切れの商品は下げたい（**属性による調整**）
- 同じくらい関連するなら、安い方／新しい方を上に出したい（**数値・時間による調整**）

これらを扱う道具が本章の3つです。役割で整理すると次のようになります。

```
              ┌─────────────────────────────────────────────┐
              │  最終スコア = 関連度スコア × ビジネス調整      │
              └─────────────────────────────────────────────┘
                        ▲                       ▲
                        │                       │
        ┌───────────────┴───────┐     ┌─────────┴───────────────┐
        │ boost / multi_match    │     │ function_score           │
        │ 「どのクエリ・フィールド │     │ 「スコアを関数で           │
        │  をどれだけ効かせるか」  │     │  後から曲げる」           │
        └────────────────────────┘     └──────────────────────────┘
```

- **boost / multi_match**＝クエリの構成段階で、語やフィールドの寄与を調整する
- **function_score**＝いったん出たスコアを、値や条件に応じて関数で変形する

## 8.2　boost ― クエリに重みをつける

`boost` は、あるクエリ節のスコアへの寄与を**相対的に**強める（または弱める）係数です。

### 具体例：商品名の一致を説明文より重視する

`bool` の `should`（OR的にスコアを足し合わせる節）で、同じ語を2つのフィールドに投げ、`name` 側だけ `boost` を上げます。

```json
GET /products/_search
{
  "query": {
    "bool": {
      "should": [
        { "match": { "name":        { "query": "軽量", "boost": 3 } } },
        { "match": { "description": { "query": "軽量", "boost": 1 } } }
      ]
    }
  }
}
```

「軽量」が **name に入っている商品**（`ノートパソコン`）は、`description` だけに入っている商品より高いスコアになります。`boost: 3` は「name の一致を 3 倍重く扱う」という意味です。

> **コラム：boost は「絶対値」ではなく「相対値」**
> `boost: 3` にしても、スコアが機械的に 3 倍になるわけではありません。BM25 の各項に係数として掛かるため、他の節との**バランス**で効き方が変わります。よくある誤解は「boost を 100 にすれば必ず最上位」ですが、複数条件が絡むと直感どおりにならないことがあります。**まず小さい値（2〜4程度）で試し、`_score` の変化を見ながら調整する**のが実務のコツです。
>
> また、かつて存在した「インデックス時 boost（マッピングでフィールドに固定の重みを付ける）」は非推奨・廃止済みです。重み付けは**クエリ時に指定する**のが現在の作法です。

## 8.3　multi_match ― 複数フィールドをまとめて検索

前節の `bool` + `should` は動きますが、フィールドが増えるほど冗長です。同じ語を複数フィールドに投げるなら `multi_match` が簡潔です。

### 基本形

```json
GET /products/_search
{
  "query": {
    "multi_match": {
      "query": "軽量",
      "fields": ["name", "description"]
    }
  }
}
```

### フィールド単位の重み付け（キャレット記法）

`fields` の各要素に `^数値` を付けると、そのフィールドの重みを指定できます。8.2 の `bool` 版と同じ意図を、1行で書けます。

```json
{
  "multi_match": {
    "query": "軽量",
    "fields": ["name^3", "description"]
  }
}
```

### 5つの type ― スコアの合成方法が変わる

`multi_match` の挙動は `type` で決まります。主要な5つを押さえましょう。

| type | 何をするか | 向いている場面 |
|---|---|---|
| `best_fields`（既定） | 各フィールドで個別にスコア計算し、**最も高い1フィールド**のスコアを採用 | 「name か description の**どちらか**によく合う文書」を上位に |
| `most_fields` | 各フィールドのスコアを**合算** | 同じ内容を**別アナライザー**で持つ multi-fields（例：`name` と `name.ngram`）を足し合わせる |
| `cross_fields` | 複数フィールドを**1つの大きなフィールド**とみなして語単位で評価 | 姓と名、都道府県と市区町村など、**1つの情報が複数フィールドに分かれている**場合 |
| `phrase` | `best_fields` と同じ合成だが、各フィールドで**フレーズ一致**（語順を考慮） | 「軽量 ノートパソコン」を語順込みで探す |
| `phrase_prefix` | `phrase` の末尾語を**前方一致**にする | 検索ボックスのサジェスト用途 |

**best_fields と most_fields の違い（図）**

「軽量」が name と description の両方にある文書を考えます。

```
best_fields:   score = max( name_score , description_score )
               ＝ 一番よく当たったフィールドを代表値にする（他はtie_breakerで一部だけ加算可）

most_fields:   score = name_score + description_score
               ＝ 当たったフィールドが多いほど高くなる
```

### tie_breaker

`best_fields` は既定で「最良の1フィールドだけ」を見ますが、`tie_breaker`（0〜1）を指定すると、**他フィールドのスコアも一部だけ加算**できます。

```json
{
  "multi_match": {
    "query": "防水",
    "fields": ["name^3", "description"],
    "type": "best_fields",
    "tie_breaker": 0.3
  }
}
```

「基本は name の一致を重視しつつ、description にも当たっていれば少し加点」という、実務でよく使うバランス設定です。

> **コラム：typeの選び方（迷ったら）**
> - まず `best_fields`（既定）でよい。多くの商品検索はこれで十分。
> - 「フィールドをまたいで語を数えたい」なら `cross_fields`。ただし各フィールドのアナライザーが揃っている必要がある点に注意。
> - `most_fields` は「同じテキストを違う解析でインデックスした multi-fields」とセットで真価を発揮する。単に別々のフィールドを足したいだけなら意図とズレやすい。

## 8.4　function_score ― スコアを関数で調整する

`boost` / `multi_match` が「クエリの作り方」でスコアを調整するのに対し、`function_score` は**いったん計算されたスコアを、関数で後から変形**します。「関連度は保ちつつ、価格・在庫・新しさなどのビジネス要因を混ぜたい」ときの主役です。

### 全体構造

```json
GET /products/_search
{
  "query": {
    "function_score": {
      "query":     { /* まず普通に検索するクエリ（BM25スコアの元） */ },
      "functions": [ /* スコアを変形する関数のリスト */ ],
      "score_mode": "sum",       // 複数関数どうしの合成方法
      "boost_mode": "multiply",  // 関数スコアと query スコアの合成方法
      "max_boost":  10,          // 関数スコアの上限（任意）
      "min_score":  0.5          // これ未満の文書を除外（任意）
    }
  }
}
```

### スコアの流れ（図）

```
 query（例: multi_match）
      │  → 文書ごとに BM25 スコア S_q を算出
      ▼
 functions[] を評価
   ├ 関数1 → f1
   ├ 関数2 → f2   ─┐ score_mode で合成
   └ 関数3 → f3   ─┘（multiply/sum/avg/max/min/first） → F
      ▼
 boost_mode で S_q と F を合成
   （multiply/replace/sum/avg/max/min）
      ▼
 最終スコア
```

- **score_mode**：関数どうしをどう束ねるか（既定 `multiply`）
- **boost_mode**：束ねた関数スコア `F` を、検索本体のスコア `S_q` とどう合わせるか（既定 `multiply`）

特に `boost_mode: "replace"` は「BM25 を無視して関数スコアだけを最終スコアにする」ので、**全文検索の関連度が反映されなくなる**点に注意してください。

### 主な関数

#### (1) weight ＋ filter ― 条件付きの単純な倍率

「在庫あり（`in_stock: true`）の商品を優遇する」例です。`filter` に一致した文書にだけ `weight` が掛かります。

```json
GET /products/_search
{
  "query": {
    "function_score": {
      "query": {
        "multi_match": { "query": "スピーカー", "fields": ["name^3", "description"] }
      },
      "functions": [
        { "filter": { "term": { "in_stock": true } }, "weight": 2 }
      ],
      "score_mode": "sum",
      "boost_mode": "multiply"
    }
  }
}
```

#### (2) field_value_factor ― 数値フィールドをスコアに反映

数値フィールドの値そのものをスコアに使います。「価格が高いほど（プレミアム商品ほど）上位」にする例です。

```json
{
  "function_score": {
    "query": { "match": { "description": "スピーカー" } },
    "field_value_factor": {
      "field": "price",
      "modifier": "log1p",
      "factor": 0.001,
      "missing": 1
    },
    "boost_mode": "multiply"
  }
}
```

- `modifier`：値の変換（`log1p`, `ln1p`, `sqrt`, `square`, `reciprocal` など）。生の価格をそのまま掛けると桁が大きすぎて暴れるため、`log1p` などで**なだらか**にするのが定石。
- `factor`：変換前に掛ける係数。
- `missing`：フィールドが無い文書に使う既定値（**計算エラーの回避に重要**）。

#### (3) 減衰関数（decay）― 「ある値の近さ」でスコアを付ける

`gauss` / `linear` / `exp` の3種類。数値・日付・地理座標に使え、「基準（origin）に近いほど高く、離れるほど減衰」を表現します。「5,000円前後の商品を優先」する例です。

```json
{
  "function_score": {
    "query": { "match_all": {} },
    "functions": [
      {
        "gauss": {
          "price": {
            "origin": 5000,
            "scale":  3000,
            "offset": 500,
            "decay":  0.5
          }
        }
      }
    ]
  }
}
```

- `origin`：最高スコアになる中心値（ここでは5,000円）
- `offset`：origin からこの範囲内は減衰しない（±500円は満点）
- `scale` と `decay`：origin から `scale` だけ離れた点でスコアが `decay` 倍になる（ここでは 8,000円で 0.5 倍）

> 日付フィールド（例：`created_at`）に `gauss` を使えば「新しいほど上位」も同じ仕組みで書けます。フェーズ後半で扱うログ検索でも頻出のパターンです。

#### (4) random_score ― ランダムに散らす

同点の文書をシャッフルしたいときに使います。`field` に `_seq_no` を指定すると、同じ `seed` で結果が安定します（ページングでの並び崩れを防ぐ）。

```json
{
  "function_score": {
    "query": { "match_all": {} },
    "random_score": { "seed": 10, "field": "_seq_no" }
  }
}
```

### score_mode と boost_mode（早見表）

| モード | `score_mode`（関数どうし） | `boost_mode`（関数 vs クエリ） |
|---|---|---|
| `multiply` | 各関数スコアを掛ける（既定） | 掛ける（既定） |
| `sum` | 足す | 足す |
| `avg` | 平均 | 平均 |
| `max` / `min` | 最大 / 最小 | 最大 / 最小 |
| `first` | 最初にマッチした関数のみ | ― |
| `replace` | ― | クエリスコアを無視し関数スコアで置換 |

> **コラム：function_score の“新しめ”の代替手段**
> `function_score` は強力ですが、用途が単純なら専用クエリの方が高速・簡潔なことがあります。
> - **`script_score` クエリ**：スクリプトで自由にスコアを計算。`field_value_factor` 的な計算を柔軟に書ける。
> - **`rank_feature` クエリ ＋ `rank_feature` 型フィールド**：人気度・PageRank のような「大きいほど良い」数値での加点に特化し、大規模でも効率的。
> - **`distance_feature` クエリ**：日付や位置の「近さ」加点に特化（減衰関数の軽量版）。
>
> まずは概念理解のため `function_score` を使いこなし、パフォーマンスが問題になったらこれらを検討する、という順序がおすすめです。

## 8.5　実践：3つを組み合わせる

最後に、本章の道具を1つのクエリにまとめます。要件は次のとおり。

1. **name と description** を対象に「軽量 ワイヤレス」を検索（name を重視）＝ `multi_match` + キャレット
2. **在庫あり**を優遇 ＝ `weight` + `filter`
3. **10,000円前後**を優遇 ＝ `gauss`
4. 関連度（BM25）は保ちつつ、上記の調整を掛け合わせる ＝ `boost_mode: multiply`

```json
GET /products/_search
{
  "query": {
    "function_score": {
      "query": {
        "multi_match": {
          "query": "軽量 ワイヤレス",
          "fields": ["name^3", "description"],
          "type": "best_fields",
          "tie_breaker": 0.3
        }
      },
      "functions": [
        { "filter": { "term": { "in_stock": true } }, "weight": 1.5 },
        {
          "gauss": {
            "price": { "origin": 10000, "scale": 5000 }
          }
        }
      ],
      "score_mode": "sum",
      "boost_mode": "multiply"
    }
  }
}
```

読み解き方：

- `multi_match` が各文書の**関連度スコア S_q** を出す（name の一致は3倍重い）。
- `functions` の2つを `score_mode: sum` で足して**関数スコア F** を作る（在庫ありなら +1.5、価格が10,000円に近いほど大きい値）。
- `boost_mode: multiply` で **最終スコア = S_q × F**。関連度が低い文書は F をいくら盛っても上位に来にくく、「関連度を土台にビジネス調整を乗せる」バランスになります。

`_explain` API（`GET /products/_search { "explain": true, ... }`）を付けると、各文書のスコアがどの節・関数から来たか内訳を確認できます。調整の効き方を確かめながら値を詰めていきましょう。

## まとめ

- **boost** はクエリ節の寄与を**相対的に**調整する係数。絶対倍率ではないので、小さい値から `_score` を見て調整する。インデックス時 boost は廃止済み。
- **multi_match** は複数フィールドの検索を簡潔に書く。`^` でフィールド重み、`type`（`best_fields` / `most_fields` / `cross_fields` / `phrase` / `phrase_prefix`）で合成方法が変わる。迷ったら `best_fields` + `tie_breaker`。
- **function_score** はスコアを関数で変形する。`weight`＋`filter`、`field_value_factor`、減衰関数（`gauss`/`linear`/`exp`）、`random_score` などを、`score_mode` で束ね `boost_mode` で本体スコアと合成する。
- `boost_mode: replace` は関連度を捨てるので要注意。基本は `multiply` で「関連度 × ビジネス調整」。
- 単純用途では `script_score` / `rank_feature` / `distance_feature` クエリも選択肢。

## 理解度チェック

**Q1.** `multi_match` で `fields: ["name^3", "description"]`、`type: "best_fields"` としたとき、スコアはどう決まりますか。

<details><summary>解答</summary>

name と description で**個別にスコアを計算し、最も高い1フィールドのスコア**を代表値として採用します（name は重み3倍で効く）。`tie_breaker` を指定すると、他フィールドのスコアも一部だけ加算されます。`most_fields` のように全フィールドを合算するわけではない点がポイントです。
</details>

**Q2.** `function_score` で `boost_mode: "replace"` にすると何が起きますか。実務上の注意点は。

<details><summary>解答</summary>

検索本体（BM25）のスコアを**無視**し、関数スコアだけが最終スコアになります。価格や在庫だけで並び、**全文検索の関連度が反映されなくなる**ため、「関連度は無視してよい」用途以外では避けます。関連度を土台に調整を乗せたいなら `multiply`（既定）を使います。
</details>

**Q3.** 「在庫あり（`in_stock: true`）の商品だけスコアを2倍にしたい」。`function_score` でどう書きますか。

<details><summary>解答</summary>

`functions` に `filter` で条件を絞り、`weight` を付けます。

```json
"functions": [
  { "filter": { "term": { "in_stock": true } }, "weight": 2 }
]
```

`filter` に一致した文書にだけ `weight` が適用されます。最終的な効き方は `score_mode` / `boost_mode` の設定によって決まります。
</details>

**Q4.** 「5,000円前後の商品を優先」したい。どの関数を使い、主要パラメータは何ですか。

<details><summary>解答</summary>

**減衰関数**（`gauss` など）を使います。主要パラメータは、中心値の `origin`（=5000）、満点となる許容幅 `offset`、減衰の速さを決める `scale` と `decay`。`origin` から `scale` だけ離れた点でスコアが `decay` 倍になります。
</details>

**Q5.** `field_value_factor` で `missing` を指定するのはなぜですか。

<details><summary>解答</summary>

対象フィールドの値を持たない文書があると、計算が破綻したり例外になったりします。`missing` は**そのフィールドが無い文書に使う既定値**で、エラーを避けて安定した挙動にするために指定します。
</details>

## 次章予告

第9章「**シノニム（類義語）辞書**」では、「PC」と「パソコン」、「イヤホン」と「ヘッドホン」のように、**表記や語は違うが意味は同じ**語をまとめてヒットさせる仕組みを扱います。`synonym_graph` トークンフィルタの設定、Elasticsearch の **Synonyms API（synonym set）** による辞書管理、そして「**インデックス時に展開するか／検索時に展開するか**」の設計トレードオフを、`products` インデックスで実際に試します。本章の重み付けと合わせることで、「ゆるく拾って、賢く並べる」検索に近づきます。
