## この章で学ぶこと（学習目標）

この章を読み終えると、次のことができるようになります。

- 検索応答の構造（`hits`・`_score`・`_source`）を読める
- **`match`（全文検索）** と **`term`（完全一致）** を正しく使い分けられる
- **`range`** で数値・日付の範囲検索ができる
- **`bool` クエリ**（must / should / filter / must_not）で複合条件を組み立てられる
- **`filter` と `must` の違い**（スコアの有無・速度）を理解して使い分けられる

第5章までで検索対象のデータが揃いました。この章は**検索の本丸**です。第3章で学んだ「text × match」「keyword × term」が、ここで実際に動くクエリになります。

## 6.1 Query DSLとは

Elasticsearchの検索は、**`_search` エンドポイントにJSONで問い合わせ**を送る形で行います。このJSONで書く検索言語を **Query DSL（Domain Specific Language）** と呼びます。基本の形はこうです。

```
GET /products/_search
{
  "query": {
    ここに検索条件を書く
  }
}
```

`query` の中身を差し替えることで、さまざまな検索を表現します。以降、Kibana Dev Tools で実行しながら読み進めてください。

---

## 6.2 まず全件取得して、応答の読み方を知る

検索条件を付けず、全ドキュメントを取得してみます。

```
GET /products/_search
{
  "query": { "match_all": {} }
}
```

応答（抜粋）はこうなります。

```json
{
  "hits": {
    "total": { "value": 6 },
    "max_score": 1.0,
    "hits": [
      {
        "_id": "1",
        "_score": 1.0,
        "_source": { "name": "軽量ノートパソコン", "price": 98000, ... }
      },
      ...
    ]
  }
}
```

応答の読み方の基本は次の3つです。

- **`hits.total.value`** … 条件に一致した件数（ここでは6件）
- **`hits.hits`** … 実際にヒットしたドキュメントの配列
- **`_score`** … 関連度スコア（第1章で学んだ「関連の強さ」。大きいほど上位）

`match_all` は全件一致なので、スコアはどれも同じ（1.0）です。次の `match` からは、このスコアが意味を持ち始めます。

> 📌 **件数制限**：既定では上位10件だけ返ります。件数は `"size": 20` で変えられます。`"from": 10` と組み合わせるとページ送り（ページネーション）ができます。

## 6.3 `match` — 全文検索

**`match`** は、text型フィールドに対する**全文検索**のクエリです。第1章・第4章で学んだ「検索キーワードも、文書と同じアナライザーで分割してから照合する」を実行します。

```
GET /products/_search
{
  "query": {
    "match": { "name": "ノートパソコン" }
  }
}
```

→ 「ノートパソコン」を含む商品、つまり **1（軽量ノートパソコン）・2（ゲーミングノートパソコン）・6（ノートパソコン用スタンド）** がヒットします。

なぜ「軽量ノートパソコン」が「ノートパソコン」でヒットするのか。第4章を思い出してください。`name` はkuromojiで `軽量 / ノート / パソコン` のように分割されて索引されており、検索語「ノートパソコン」も同じく `ノート / パソコン` に分割されて照合されるからです。**両側で同じアナライザーを通す**——これが全文検索の要でした。

### スコアで並ぶ

結果は `_score` の高い順に並びます。検索語との関連が強い（語がよく一致する・珍しい語で一致する）ものほど上位に来ます。これが `LIKE` にはなかった、全文検索エンジンの強みです。

### 複数語の指定

`match` に複数語を渡すと、既定では **どれか（OR）** を含むものがヒットします（一致が多いほどスコアが上がります）。

```
GET /products/_search
{
  "query": {
    "match": {
      "description": "ゲーム 高性能"
    }
  }
}
```

「すべて（AND）含む」ものだけに絞りたいときは `operator` を指定します。

```
GET /products/_search
{
  "query": {
    "match": {
      "description": { "query": "ゲーム 高性能", "operator": "and" }
    }
  }
}
```

## 6.4 `term` — 完全一致

**`term`** は、値を**分割せずそのまま**照合する、完全一致のクエリです。第3章で学んだとおり、主に **keyword型** に対して使います。

カテゴリ（keyword）で絞り込む例：

```
GET /products/_search
{
  "query": {
    "term": { "category": "周辺機器" }
  }
}
```

→ **3・4・5**（周辺機器の3商品）がヒットします。

真偽値（boolean）にも使えます。「在庫切れ（in_stock=false）」の商品：

```
GET /products/_search
{
  "query": {
    "term": { "in_stock": false }
  }
}
```

→ **4（メカニカルキーボード）** がヒットします。

### 複数の値のどれか — `terms`

複数候補のいずれかに一致、は `terms`（複数形）を使います。

```
GET /products/_search
{
  "query": {
    "terms": { "category": ["パソコン", "アクセサリ"] }
  }
}
```

→ パソコンまたはアクセサリ、つまり **1・2・6** がヒットします。

### ⚠️ よくある落とし穴（第3章の復習）

`term` を **text型フィールド**に使うと、うまくいかないことがあります。text型は分割・正規化された後のトークンで索引されているため、元の見た目どおりの文字列を `term` で指定しても一致しないのです。

**「全文検索したい → match × text」「完全一致で絞りたい → term × keyword」** を徹底しましょう。迷ったらこの対応に立ち返ってください。

---

## 6.5 `range` — 範囲検索

数値や日付の範囲は **`range`** で指定します。価格が1万円以上・10万円以下の商品：

```
GET /products/_search
{
  "query": {
    "range": {
      "price": { "gte": 10000, "lte": 100000 }
    }
  }
}
```

→ **1（98000）・4（12800）・5（45000）** がヒットします。

範囲指定のキーワードは次の4つです。

| 記号 | 意味 |
|------|------|
| `gte` | 以上（≧） |
| `gt` | より大きい（＞） |
| `lte` | 以下（≦） |
| `lt` | 未満（＜） |

日付フィールドにも同じ形で使えます（例：`"gte": "2026-01-01"`）。

---

## 6.6 `bool` — 複合条件を組み立てる

実際の検索は「キーワードで探しつつ、カテゴリと価格で絞り込む」のように**複数条件の組み合わせ**になります。これを担うのが **`bool` クエリ**です。boolは、次の4つの箱に条件を入れて組み立てます。

| 箱 | 意味 | スコアへの影響 |
|----|------|---------------|
| `must` | すべて満たす（AND） | **する**（関連度に反映） |
| `should` | できれば満たす（OR/加点） | **する**（一致すると加点） |
| `filter` | すべて満たす（AND） | **しない**（合否のみ） |
| `must_not` | 満たしてはいけない（除外） | しない |

### `must` と `filter` の違い（最重要）

`must` と `filter` は、どちらも「AND条件」で結果は同じように絞れます。違いは**スコアを計算するかどうか**です。

- **`must`**：条件との「関連の強さ」をスコアに反映する。**全文検索の本体**に使う
- **`filter`**：合否（Yes/No）だけを見てスコアに影響しない。**単なる絞り込み**に使う

そして `filter` には大きな利点があります。**スコア計算をしないぶん速く、結果がキャッシュされて再利用される**のです。だから、

> カテゴリ・価格帯・在庫の有無・タグのような「一致するか・しないか」だけの絞り込みは、`must` ではなく **`filter`** に入れるのが定石

です。「関連度で並べたい全文検索は `must`（match）」「単なる絞り込みは `filter`」と覚えてください。

### 実践：現実的な検索を組む

「**"ノートパソコン" で検索し、カテゴリはパソコン、10万円以下、在庫あり**」を組み立てます。

```
GET /products/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "name": "ノートパソコン" } }
      ],
      "filter": [
        { "term":  { "category": "パソコン" } },
        { "term":  { "in_stock": true } },
        { "range": { "price": { "lte": 100000 } } }
      ]
    }
  }
}
```

処理の流れを追うと：

1. `must` の match で「ノートパソコン」を含む **1・2・6** が候補になり、関連度でスコアされる
2. `filter` でカテゴリ=パソコンに絞る → **1・2**（6はアクセサリなので除外）
3. `filter` で在庫ありに絞る → 1・2 とも該当
4. `filter` で10万円以下に絞る → **1**（2は21万円で除外）

結果として **1（軽量ノートパソコン）** が返ります。全文検索の関連度（must）と、機械的な絞り込み（filter）が、役割分担して組み合わさっているのが分かります。

### `should` — 加点する / OR条件にする

`should` は「必須ではないが、一致したら加点したい」条件に使います。たとえば先ほどの検索に「タグが"軽量"ならより上位に」を足す：

```
GET /products/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "name": "ノートパソコン" } }
      ],
      "should": [
        { "term": { "tags": "軽量" } }
      ]
    }
  }
}
```

→ ヒットは **1・2・6**（mustの結果）ですが、タグに「軽量」を持つ **1・6** のスコアが上がり、上位に来ます。ランキングを調整したいときの基本テクニックです。

> 📌 **補足**：`must` や `filter` がなく `should` だけの場合は、「**最低1つは満たす**」が既定の条件になります（OR検索のように働きます）。この最低個数は `minimum_should_match` で調整できます。

### `must_not` — 除外する

「在庫切れは除く」なら `must_not` に入れます。

```
GET /products/_search
{
  "query": {
    "bool": {
      "must":     [ { "match": { "description": "ゲーム" } } ],
      "must_not": [ { "term":  { "in_stock": false } } ]
    }
  }
}
```

「ゲーム」を説明に含むのは **2・4** ですが、在庫切れの **4** が除外され、**2** だけが残ります。

## 6.8 まとめ

- 検索は `_search` に Query DSL（JSON）で問い合わせる。応答は `hits`・`_score`・`_source` を読む
- **`match`＝全文検索**（text型、キーワードも分割して照合、スコアで並ぶ）
- **`term`＝完全一致**（keyword型・boolean。複数候補は `terms`）
- **`range`** で数値・日付の範囲検索（`gte`/`gt`/`lte`/`lt`）
- **`bool`** で複合条件：`must`(AND・加点) / `should`(加点・OR) / `filter`(AND・合否のみ) / `must_not`(除外)
- **全文検索の本体は `must`(match)、機械的な絞り込みは `filter`**（速く・キャッシュされる）
- 迷ったら「match × text」「term × keyword」に立ち返る

これで、Elasticsearchで実用的な検索が書けるようになりました。

## 6.9 理解度チェック

**問1.** `match` と `term` は、それぞれどんな検索で、主にどの型のフィールドに使いますか。

**問2.** `bool` の `must` と `filter` は、どちらもAND条件ですが、決定的な違いは何ですか。絞り込み条件はどちらに入れるのが定石ですか。

**問3.** 価格が3,000円「より高く」20,000円「以下」の商品を探す `range` を書いてください。

**問4.** 「説明文に"高性能"を含み、かつ在庫ありで、パソコンカテゴリは除外する」を `bool` で組み立ててください。

<details>
<summary>解答を見る</summary>

**問1.** `match` は全文検索で、主に text 型に使う（キーワードを分割して照合し、関連度スコアで並ぶ）。`term` は完全一致で、主に keyword 型（や boolean）に使う（値を分割せずそのまま照合）。

**問2.** `must` はスコア計算をして関連度に反映するのに対し、`filter` はスコアに影響せず合否だけを見る（そのぶん速く、結果がキャッシュされる）。カテゴリ・価格帯・在庫のような単なる絞り込みは `filter` に入れるのが定石。

**問3.**
```
{ "range": { "price": { "gt": 3000, "lte": 20000 } } }
```

**問4.**
```
GET /products/_search
{
  "query": {
    "bool": {
      "must":     [ { "match": { "description": "高性能" } } ],
      "filter":   [ { "term":  { "in_stock": true } } ],
      "must_not": [ { "term":  { "category": "パソコン" } } ]
    }
  }
}
```
（「高性能」を説明に含むのは2・5。うち在庫ありで、パソコンカテゴリの2を除外 → 5 が残る）

**ハンズオン解答例**
1. `bool` の filter に `term category=周辺機器`・`range price lte 5000`・`term in_stock=true` → 3
2. `match description=軽量`（関連度順に自然に並ぶ） → 1・3・6 など
3. `bool` の filter に `term tags=高性能`、must_not に `range price gte 200000` → 5（2は21万円で除外）

</details>
