## この章で学ぶこと（学習目標）

この章を読み終えると、次のことができるようになります。

- ElasticsearchのREST APIで、ドキュメントを**登録・取得・更新・削除（CRUD）** できる
- `PUT` による更新が**丸ごと置き換え**になるという落とし穴を避けられる
- `_update` で**部分更新**ができる
- **Bulk API** で大量データを一括投入できる（そのための **NDJSON** 形式を理解する）
- なぜ登録直後に検索してもヒットしないことがあるのか（**準リアルタイム**）を説明できる
- 次章の検索練習に使う**サンプルデータを自分で用意**できる

第4章までで、データの構造（マッピング）と日本語の索引化（kuromoji）が整いました。この章では、そこへ**実際にデータを出し入れ**します。

---

## 5.1 CRUDとREST API — 全体像

**CRUD** とは、データ操作の基本4種、Create（作成）/ Read（取得）/ Update（更新）/ Delete（削除）の頭文字です。Elasticsearchでは、これらを**HTTPメソッド**で表現します。第2章で「ElasticsearchはHTTP/JSONで操作する」と述べましたが、その具体形がこれです。

| 操作 | HTTPメソッド | 主なエンドポイント |
|------|------------|-------------------|
| Create（作成） | `PUT` / `POST` | `PUT /インデックス/_doc/ID` |
| Read（取得） | `GET` | `GET /インデックス/_doc/ID` |
| Update（更新） | `POST` | `POST /インデックス/_update/ID` |
| Delete（削除） | `DELETE` | `DELETE /インデックス/_doc/ID` |

URLの基本形は **`/インデックス名/_doc/ドキュメントID`** です。「どの箱の、どのIDのドキュメントか」を表します。以降の例はすべて Kibana Dev Tools で実行できます。

---

## 5.2 Create — ドキュメントの登録

### IDを自分で指定して登録する（PUT）

```
PUT /products/_doc/1
{
  "name": "軽量ノートパソコン",
  "category": "パソコン",
  "price": 98000
}
```

実行すると、次のような応答が返ります（抜粋）。

```json
{
  "_index": "products",
  "_id": "1",
  "_version": 1,
  "result": "created"
}
```

- `result: "created"` … 新規作成された合図
- `_version: 1` … このドキュメントのバージョン（更新のたびに増える）

### IDをElasticsearchに任せる（POST）

IDを指定せず `POST` を使うと、Elasticsearchが**IDを自動採番**します。ログのように「1件1件を区別できればIDは何でもよい」ケースで便利です。

```
POST /products/_doc
{
  "name": "ワイヤレスマウス",
  "category": "周辺機器",
  "price": 3200
}
```

応答の `_id` に、自動生成された文字列（例：`x8Kd...`）が入ります。

> 💡 **使い分け**：元データに主キー（商品IDなど）があるなら、それを `_id` に指定するのが定石です。そうすれば「同じ商品を二重登録しない」「後で更新・削除しやすい」といった利点があります。

## 5.3 Read — ドキュメントの取得

IDを指定して1件取得します。

```
GET /products/_doc/1
```

```json
{
  "_index": "products",
  "_id": "1",
  "_found": true,
  "_source": {
    "name": "軽量ノートパソコン",
    "category": "パソコン",
    "price": 98000
  }
}
```

登録した元のJSONは **`_source`** に入って返ってきます（第3章で触れたメタデータの一つです）。存在しないIDを取得すると `found: false` が返ります。

> 📌 **補足**：IDでの1件取得（GET）は**即時**に反映されます。一方、検索（次章の `_search`）は少し事情が違います。これは5.7で説明します。

---

## 5.4 Update — ドキュメントの更新

ここに**初学者が必ずハマる落とし穴**があります。

### ⚠️ 落とし穴：PUTは「丸ごと置き換え」

すでに存在するID `1` に対して、価格だけ変えるつもりで次を実行するとどうなるでしょうか。

```
PUT /products/_doc/1
{
  "price": 88000
}
```

一見「priceだけ更新」に見えますが、**実際にはドキュメント全体が置き換わります**。`name` も `category` も消え、`price` だけのドキュメントになってしまいます。`PUT /_doc/ID` は「そのIDの中身を、送ったJSONで丸ごと上書きする」動作だからです。

応答の `result` は `updated`、`_version` は `2` に上がりますが、中身は失われています。これは非常によくある事故です。

### 正しい部分更新：`_update` を使う

一部のフィールドだけ更新したいときは、**`_update` エンドポイント**を使い、`doc` に**変更したいフィールドだけ**を書きます。

```
POST /products/_update/1
{
  "doc": {
    "price": 88000
  }
}
```

これなら `name` や `category` は保持されたまま、`price` だけが更新されます。**「部分更新は `_update` と `doc`」** とセットで覚えてください。

### 存在しなければ作る：upsert（応用）

「あれば更新、なければ新規作成」をまとめて行いたいときは `upsert` を使います（詳細は必要になったときに調べれば十分です）。

```
POST /products/_update/1
{
  "doc": { "price": 88000 },
  "upsert": { "name": "新商品", "category": "未分類", "price": 88000 }
}
```

### コラム：ドキュメントは内部的には「書き換えていない」

Elasticsearchの更新は、実は既存データをその場で書き換えているのではありません。内部的には**古いドキュメントを「削除済み」と印を付け、新しい内容で作り直して**います（だから `_version` が増えます）。

これは第1章で学んだ**転置インデックスの性質**によるものです。索引を部分的に書き換えるのは難しいため、「作り直す」方式を取っているのです。頻繁な更新が多いデータでは、この特性を頭の隅に置いておくと、後々の運用（フェーズ6）で役立ちます。

---

## 5.5 Delete — ドキュメントの削除

IDを指定して1件削除します。

```
DELETE /products/_doc/1
```

`result: "deleted"` が返れば成功です。

### 条件に合うものをまとめて削除（応用）

「在庫切れの商品をすべて削除」のように、条件で複数削除したいときは **Delete By Query** を使います（クエリの書き方は次章で学びます）。

```
POST /products/_delete_by_query
{
  "query": {
    "term": { "in_stock": false }
  }
}
```

## 5.6 Bulk API — 大量データの一括処理

ここまでは1件ずつの操作でした。しかし、数千・数万件を1件ずつ登録すると、**リクエストの往復（通信）が多すぎて非常に遅く**なります。そこで、複数の操作を**1回のリクエストにまとめて送る** `_bulk` を使います。

### NDJSON という独特な形式

Bulk APIは、通常のJSONではなく **NDJSON（Newline Delimited JSON）** という形式を使います。これは「**1行1JSON**」で、改行で区切る形式です。

`index`（登録）操作は、**2行1組**で書きます。

```
{ "index": { "_index": "products", "_id": "1" } }   ← 1行目：何をするか（アクション行）
{ "name": "軽量ノートパソコン", "price": 98000 }        ← 2行目：中身（データ行）
```

- **アクション行**：操作の種類（`index` / `create` / `update` / `delete`）と対象を指定
- **データ行**：登録・更新するドキュメントの中身（`delete` の場合はデータ行なし）

これを必要な件数だけ繰り返します。

### 実行例

```
POST /_bulk
{ "index": { "_index": "products", "_id": "10" } }
{ "name": "USBハブ", "category": "周辺機器", "price": 1500 }
{ "index": { "_index": "products", "_id": "11" } }
{ "name": "webカメラ", "category": "周辺機器", "price": 4800 }
{ "delete": { "_index": "products", "_id": "99" } }
```

1回のリクエストで、2件の登録と1件の削除がまとめて処理されます。

### ⚠️ Bulkでハマりやすい点

- **JSONを整形（pretty）してはいけない**：各JSONは必ず**1行に収める**。改行で複数行にすると壊れます
- **最後の行の後にも改行が必要**：ファイルで送る場合、末尾に空行（改行）を入れる
- Dev Toolsのエディタは、この形式をそのまま貼り付けて実行できます

## 5.7 ハンズオン — 次章で使うサンプルデータを投入する

次章（Query DSL）の検索練習に向けて、日本語のサンプル商品データを用意します。**第3章のマッピング設計**と**第4章のkuromoji**を組み合わせた、実用的なインデックスを作ります。

### ステップ1：インデックスを作る（マッピング付き）

```
PUT /products
{
  "mappings": {
    "properties": {
      "name":        { "type": "text", "analyzer": "kuromoji" },
      "description": { "type": "text", "analyzer": "kuromoji" },
      "category":    { "type": "keyword" },
      "tags":        { "type": "keyword" },
      "price":       { "type": "integer" },
      "in_stock":    { "type": "boolean" }
    }
  }
}
```

> 💡 `tags` には配列（複数のタグ）が入りますが、**配列専用の型はありません**。要素の型（ここでは `keyword`）を指定しておけば、その型の値を複数持てます。これがElasticsearchでの配列の扱い方です。

> ⚠️ 5.2〜5.5の練習で既に `products` を作っている場合は、いちど `DELETE /products` で消してから、上のマッピングで作り直してください。

### ステップ2：Bulkでデータを投入する

Dev Tools に次を貼り付けて実行します。

```

{ "index": { "_id": "1" } }

{ "index": { "_id": "3" } }
{ "name": "ワイヤレスマウス", "description": "手にフィットする軽量なワイヤレスマウス。電池持ちがよく静音設計です。", "category": "周辺機器", "tags": ["軽量","ワイヤレス"], "price": 3200, "in_stock": true }

## 5.8 実務メモ：なぜ登録直後に検索してもヒットしないことがあるか

Bulk投入の直後に検索（次章の `_search`）を実行すると、**まだヒットしない**ことがあります。バグではありません。Elasticsearchが **準リアルタイム（near real-time）** で動作するためです。

- 登録したドキュメントが**検索可能になるまで、既定で約1秒**かかります
- これは、検索を速くするための内部処理（リフレッシュ）の都合です
- IDでの取得（`GET /_doc/ID`）は即時ですが、**検索（`_search`）は約1秒遅れ**ると理解しておきましょう

普通に使う分には1秒待てば済む話ですが、「入れた直後に検索したら0件で焦る」というのはよくある初学者の混乱ポイントなので、仕組みとして知っておいてください。

> 📌 テストなどで即時反映させたいときは `?refresh` を付ける方法もありますが、常用すると性能が落ちるため、通常は既定のまま（1秒待つ）で問題ありません。

## 5.9 まとめ

- CRUDはHTTPメソッドで表現：**作成=PUT/POST、取得=GET、更新=POST(_update)、削除=DELETE**
- `PUT /_doc/ID` は**丸ごと置き換え**。部分更新は必ず **`_update` + `doc`**
- ドキュメントは内部的には書き換えず「作り直し」（転置インデックスの性質による）
- 大量投入は **Bulk API**。形式は **NDJSON（1行1JSON、整形禁止、末尾に改行）**
- `index` 操作は**アクション行＋データ行の2行1組**、`delete` はアクション行のみ
- 検索は **準リアルタイム（約1秒遅れ）**。取得(GET)は即時
- 第3章のマッピング＋第4章のkuromojiで、次章用の `products` データを構築した

## 5.10 理解度チェック

**問1.** 既存のドキュメント（ID=1）の `price` だけを変えたいとき、`PUT /products/_doc/1` に `{ "price": 88000 }` を送ると何が起きますか。正しくはどうすべきですか。

**問2.** Bulk APIが使う「NDJSON」とはどんな形式ですか。通常のJSONと違って気をつける点を1つ挙げてください。

**問3.** ドキュメントを登録した直後に検索してもヒットしないことがあります。なぜですか。

**問4.** `index` 操作と `delete` 操作では、Bulkでの書き方にどんな違いがありますか。

<details>
<summary>解答を見る</summary>

**問1.** ドキュメント全体が丸ごと置き換わり、`name` や `category` が消えて `price` だけのドキュメントになってしまう。正しくは `POST /products/_update/1` に `{ "doc": { "price": 88000 } }` を送り、部分更新する。

**問2.** 「1行1JSON」で改行区切りの形式（Newline Delimited JSON）。気をつける点：各JSONを整形（pretty）せず必ず1行に収める／末尾にも改行が必要、など（いずれか1つ）。

**問3.** Elasticsearchが準リアルタイムで、登録から検索可能になるまで既定で約1秒かかるため（IDでのGET取得は即時だが、_searchは約1秒遅れる）。

**問4.** `index` は「アクション行＋データ行」の2行1組で書く。`delete` は対象を指定するアクション行のみで、データ行は書かない。

</details>
