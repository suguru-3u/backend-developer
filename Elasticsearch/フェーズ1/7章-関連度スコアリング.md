この章を読み終えると、次のことが説明できるようになります。

第6章で `match` を使ったとき、結果が `_score` の順に並びました。この章では、その**スコアの正体**を解き明かします。ここが分かると、「なぜこの記事が上に来るのか」を説明でき、次のフェーズで検索を改善する土台になります。

## 7.1 復習：スコアで並ぶということ

第1章で、全文検索エンジンは「ヒットするかしないか（◯×）」だけでなく、各ドキュメントに**関連度スコア**という点数をつけ、**点数の高い順に並べて返す**と学びました。第6章の `match` の結果で見た `_score` が、まさにそれです。

```json
"hits": [
  { "_id": "1", "_score": 2.31, "_source": {...} },
  { "_id": "6", "_score": 1.87, "_source": {...} }
]
```

問題は、**この 2.31 や 1.87 がどうやって決まるのか**です。答えの中心にあるのが **BM25** というアルゴリズムです。まずは、その中身を数式なしの直感で理解しましょう。

---

## 7.2 スコアを決める3つの要素（直感で理解する）

検索語とドキュメントの「関連の強さ」は、おおまかに次の3つで決まります。

### 要素① 出現回数（TF：Term Frequency）

**検索語が、その文書に何回出てくるか。** 多いほど「その話題について書かれている」可能性が高く、スコアが上がります。

- 「猫」を10回含む記事と、1回しか含まない記事なら、前者のほうが猫に関連が深そう

ただし重要な特徴があります。**回数が増えるほどの効果はだんだん頭打ちになります（飽和）**。1回→2回では大きく上がりますが、100回→101回ではほとんど変わりません。「10回も出れば十分猫の記事だ」という直感に合わせているのです。（この頭打ちが、後述するBM25の賢いところです。）

### 要素② 語の珍しさ（IDF：Inverse Document Frequency）

**その検索語が、全文書の中でどれだけ珍しいか。** 珍しい語で一致するほど、価値が高くスコアが上がります。

- 「の」「です」のような、どの文書にもある語で一致しても、ほとんど手がかりにならない
- 「量子コンピュータ」のような、一部の文書にしかない語で一致すれば、強い手がかりになる

つまり、**ありふれた語は軽く、珍しい語は重く**扱われます。これがIDFの考え方です。

### 要素③ 文書（フィールド）の長さ

**同じ1回の出現でも、短い文書での一致のほうが「濃い」** とみなされ、スコアが上がります。

- 10語しかないタイトルに「猫」が1回 → 猫がテーマの中心と言える
- 1万語の長文に「猫」が1回 → たまたま触れただけかもしれない

長い文書は、単に語数が多いぶん検索語も混ざりやすいので、その効果を薄める（正規化する）わけです。

> 💡 この3つを一言でまとめると、**「珍しい語が（②）」「たくさん出てきて（①）」「短い文書に含まれる（③）」ほど、関連が強い**、という直感です。BM25は、この直感を数式にしたものにすぎません。

---

## 7.3 BM25 — 3要素を組み合わせたアルゴリズム

**BM25** は、上の3要素を組み合わせてスコアを算出する、現在のElasticsearchの**標準の関連度アルゴリズム**です（バージョン5以降の既定）。名前は "Best Matching 25" に由来しますが、名前を覚える必要はありません。大事なのは、**BM25＝「①出現回数（頭打ちあり）× ②語の珍しさ × ③長さ補正」を計算するもの**、という理解です。

BM25が、単純な「出現回数×珍しさ」より優れている最大の点は、**要素①の頭打ち（飽和）** を持っていることです。これがないと、「猫猫猫猫猫…」と同じ語を大量に詰め込んだだけの文書が不当に上位に来てしまいます。BM25はこれを防ぎ、現実的な並びを作ります。

<details>
<summary>発展：BM25の数式（読み飛ばして構いません）</summary>

1つの検索語 q に対するスコアは、ざっくり次の形です。

```
score = IDF(q) × [ tf × (k1 + 1) ] / [ tf + k1 × (1 - b + b × dl / avgdl) ]
```

- `tf` … その文書での q の出現回数（要素①）
- `IDF(q)` … q の珍しさ（要素②）。全文書数と、q を含む文書数から計算
- `dl` … その文書（フィールド）の長さ、`avgdl` … 全文書の平均の長さ（要素③）
- `k1`（既定1.2）… 出現回数の**頭打ちの効き具合**を調整するつまみ
- `b`（既定0.75）… **長さ補正の強さ**を調整するつまみ

複数語のクエリなら、各語のスコアを**合計**します。`k1` と `b` はマッピングで調整できますが、まずは既定値で十分です。数式そのものより、「3要素をこう組み合わせている」という構造を掴めれば十分です。

</details>

## 7.4 スコアの内訳を見る — `explain`

「なぜこの点数なのか」は、推測せずに**Elasticsearchに聞けます**。`explain` を使うと、スコアの計算過程が返ります。第5〜6章で使った `products` で試しましょう。

検索全体で内訳を見るなら、`"explain": true` を付けます。

```
GET /products/_search
{
  "query": { "match": { "description": "ゲーム" } },
  "explain": true
}
```

特定の1ドキュメントに絞って見るなら、`_explain` エンドポイントを使います。

```
GET /products/_explain/2
{
  "query": { "match": { "description": "ゲーム" } }
}
```

返ってくる `_explanation` を読むと、スコアが **IDF（語の珍しさ）× TF（出現回数を長さで補正した値）** の掛け合わせで組み立てられている様子が、`idf`・`tf`・`dl`（フィールド長）といった項目とともに確認できます。7.2で学んだ3要素が、実際の数字として現れているのが分かるはずです。

> 💡 **試してみる価値がある実験**：`description` に「ゲーム」を1回だけ含む商品が複数あるとき、**説明文が短い商品のほうが上位に来やすい**はずです（要素③の効果）。`explain` でフィールド長 `dl` の値を見比べると、長さ補正が効いているのが確認できます。頭で理解した原理を、数字で裏取りする——これが理解を確実にします。

## 7.5 複数条件のスコアはどう合算されるか

第6章の `bool` クエリでは、複数の条件が組み合わさりました。そのスコアは次のように決まります。

- **`must` と `should` のスコアは合算される**：一致した条件が多いほど、また関連が強いほど、合計スコアが高くなる
- **`filter` と `must_not` はスコアに影響しない**：第6章で学んだとおり、合否だけを見る。だから「単なる絞り込みは filter」が、速度だけでなく**スコアを汚さない**という意味でも正しい
- **`should` は加点役**：`must` で候補を決め、`should` で「望ましい条件に合うものを上に押し上げる」という使い方ができる（第6章の「軽量タグで加点」がこれ）

つまり、**関連度で並べたい条件は must/should に、順位に関係ない絞り込みは filter に**、と振り分けるのが、スコアの観点からも理にかなっているのです。

---

## 7.6 スコアが意図とズレるとき — 改善（フェーズ2）への入口

BM25は優秀ですが、**そのままではビジネス上の「望ましい順番」と一致しないこと**もあります。典型例を挙げます。

- **珍しすぎる語に引っ張られる**：まれな語（誤字や特殊な型番など）で一致した文書が、IDFの効果で不自然に上位に来る
- **タイトルと本文が同じ重み**：本来「タイトルに一致」を重視したいのに、初期状態では本文の一致と同格に扱われる
- **新しさ・人気・在庫を反映したい**：BM25はテキストの関連度しか見ない。「新着を上に」「在庫切れは下に」といった、テキスト以外の要素は考慮しない

これらを調整するのが、次のフェーズの主題です。フェーズ2では、次のような手法を学びます。

- **フィールドごとの重み付け（boost）**：タイトルの一致を本文より重くする
- **`multi_match`**：複数フィールドをまとめて検索し、重みを付ける
- **`function_score`**：新しさ・人気度・数値などを使って、テキストのスコアを補正する

**BM25という「土台のスコア」を理解した今**なら、これらのチューニングが「何を、どちらに動かしているのか」を、納得して扱えるようになります。

---

## 7.7 スコアの扱いで知っておくこと

最後に、実務で誤解しやすい点を2つ。

- **スコアの絶対値には意味がない**：`_score` が 2.31 でも 8.0 でも、それ自体は良し悪しを表しません。意味があるのは**同じ検索の中での相対的な順位**だけです。別のクエリの `_score` と数値を比べても無意味です。
- **並び替えを指定するとスコアは使われない**：`sort` で価格順などを明示すると、BM25の関連度順ではなくなります（スコア計算自体を省くこともできます）。「関連度で並べたいのか、指定した値で並べたいのか」を意識して使い分けます。

---

## 7.8 まとめ

- 関連度スコアは、**①出現回数（TF・頭打ちあり）× ②語の珍しさ（IDF）× ③文書の長さ補正** で決まる
- Elasticsearchの標準アルゴリズムは **BM25**。3要素を組み合わせ、出現回数の飽和で不自然な上位表示を防ぐ
- **`explain`** でスコアの内訳（idf・tf・フィールド長）を確認できる
- `bool` では **must/should のスコアは合算、filter/must_not はスコアに影響しない**
- スコアの**絶対値に意味はなく**、同一検索内の相対順位だけが意味を持つ
- BM25だけでは足りない調整（重み付け・新しさ・人気）は、**フェーズ2のチューニング**で扱う

## 7.9 理解度チェック

**問1.** 関連度スコアを決める3つの要素を挙げ、それぞれ「大きいほどスコアが上がるのか下がるのか」を答えてください。

**問2.** 同じ検索語を1回だけ含む2つの文書があります。片方は短いタイトル、もう片方は長い本文です。どちらのスコアが高くなりやすいですか。理由も述べてください。

**問3.** `bool` クエリで、`filter` に入れた条件はスコアにどう影響しますか。それは速度以外にどんな利点になりますか。

**問4.** 「`_score` が 5.0 だから、別の検索で出た `_score` 3.0 の結果より関連が強い」と言えますか。

<details>
<summary>解答を見る</summary>

**問1.**
- ① 出現回数（TF）：多いほどスコアが上がる（ただし頭打ちあり）
- ② 語の珍しさ（IDF）：珍しいほどスコアが上がる（ありふれた語は下がる）
- ③ 文書の長さ：短いほど（同じ出現なら）スコアが上がる

**問2.** 短いタイトルのほう。同じ1回の出現でも、短い文書では検索語が相対的に「濃く」なり、長さ補正（要素③）によって高く評価されるため。

**問3.** `filter` の条件はスコアに影響しない（合否だけを判定する）。速度が速くキャッシュされる利点に加え、**スコアを汚さない**という利点がある。関連度で並べたい条件だけが素直にスコアに反映される。

**問4.** 言えない。`_score` の絶対値には意味がなく、比較できるのは同じ検索の中での相対順位だけ。別々の検索のスコアを数値で比べても意味がない。

</details>
